\chapter*{Calling Convention}
\addcontentsline{toc}{chapter}{Calling Convention}


The previous chapters introduced the register usage conventions and the overall stack layout defined by the runtime architecture. Building on that foundation, this chapter describes the mechanisms used to invoke routines, both simple procedures and more substantial functions, and the conventions that govern control transfer, parameter passing, register preservation, and stack-frame management.

The discussion begins with the simplest form of call, the local call, which transfers control between routines within the same module. Local calls illustrate the fundamental steps of the calling convention without the additional considerations required for inter-module calls.

A routine call proceeds through several well-defined stages. First, the caller evaluates all argument expressions and places the resulting values into the parameter area, either in registers or on the stack as required. After preparing the parameters, the caller saves any registers whose values must remain intact across the call. These caller-save registers must be preserved by the caller before branching to another routine.

The called routine, the callee, accesses the incoming parameters in the established layout of the parameter area and begins execution at its entry point. Upon entry, the callee allocates a stack frame, records the return link, and saves any callee-save registers it intends to overwrite. This setup ensures that both caller and callee operate independently while maintaining a consistent runtime state.

Once the callee completes its computation, it prepares to return. All callee-save registers are restored, the return address is retrieved from the frame marker, and control is transferred back to the caller. This division of responsibility allows efficient cooperation between routines and avoids unnecessary state preservation.

As described earlier, the general-purpose register set is partitioned into caller-save and callee-save regions. A compiler selects registers from these regions as needed, balancing the work of saving and restoring state between the caller and the callee. When possible, redundant saves and restores are eliminated. The runtime architecture distinguishes between two basic categories of calls: local calls, which remain within a single module, and external calls, which cross module boundaries and require additional handling. The following sections describe the two types in detail.

\section*{Local Call}
\addcontentsline{toc}{section}{Local Call}

A local call is by far the most common type of procedure call. The following code snippets illustrate such a call. The calling procedure prepares the call by loading the arguments, saving any registers that the callee might modify, and then branching to the callee. After the callee returns, the caller restores its saved registers and continues execution.

\begin{figure}[H]
\centering
\lstset{style=codesnippetstyle}
\begin{lstlisting}
; calling procedure

			...					; load arguments
			...					; save caller saves
			BL target, RL 		; branch to "target", save return
			...					; restore caller saves
\end{lstlisting}
\end{figure}

The called procedure begins by saving the return link, allocating its stack frame, and saving any registers that the caller expects to remain unchanged across the call. At the end of the routine, it restores the return address, restores any callee-saved registers, deallocates the stack frame, and finally branches back to the caller using the restored return address.

\begin{figure}[H]
\centering

\lstset{style=codesnippetstyle}
\begin{lstlisting}
; called procedure
            
target:     ...					; save RL
			... 				; allocate stack frame
			...					; save callee save
			
			...					; do the work
			
			...					; restore RL
			...					; restore callee saves
			...					: deallocate stack frame
			BV (RL) 			; return to caller
\end{lstlisting}
\end{figure}

The code above shows the full sequence of steps that may be required for a local procedure call. Depending on the specific call and the nature of the callee, some steps may be omitted. For example, if the caller does not use any caller-saved registers, there is no need to save them before the call. Likewise, a procedure with no arguments does not need to evaluate arguments.

The callee can also omit unnecessary steps. A leaf procedure, which is a procedure that does not call any other procedure, does not need to save the return address. If it does not use any callee-saved registers, those need not be saved or restored. Finally, if a leaf procedure uses no caller-saved registers, has no local variables, and does not need to save any temporary registers, it may not require a stack frame at all.

The instructions \texttt{B}, \texttt{BR} and \texttt{BV} are the instructions used for a local branch. The address computation, whether IA-relative or region relative, will not impact the region portion of the instruction address.

The first four arguments are passed via the registers ARG0 to ARG3. Nevertheless, the stack frame parameter area allocates stack space for them. Any further parameter is stored in the stack frame parameter area. Upon return, the called procedure may pass up to two return values in registers, \texttt{Ret0} and \texttt{Ret1}. Arguments and return values are addressed relative to the stack frame pointer of the calling procedure. Since the called procedure may also allocates a stack frame, the size of the callee stack frame needs to be added to the fixed offsets where the arguments are in the previous frame.

\section*{External Calls}
\addcontentsline{toc}{section}{External Calls}

An external call is defined as potentially crossing the region boundary. The \texttt{BE} instruction computes the effective address from a general register and an immediate offset encoded in the instruction. Like the local calls, this is a 32-bit unsigned address computation. In contrast to the local branch type instruction, the general register used as the base register contains the full virtual address and not only the offset portion. The \texttt{BE} is used for branching to a target in another region and also to return from that region. The following code snippets show an external call.

\begin{figure}[H]
\centering
\lstset{style=codesnippetstyle}
\begin{lstlisting}
; calling code

			...					; load arguments
			...					; save caller saves
			LD R1, -ofs(DP)	    ; obtain the target address    
			BE R1, RL    		; branch to "target", save return
			...					; restore caller saves
\end{lstlisting}
\end{figure}

Before executing the \texttt{BE} instruction, that target address is fetched from an area below the global area pointed to by the DP register. This area is referred to as the linkage table, outlined in the next section. The following figure shows the called procedure.

\begin{figure}[H]
\centering
\lstset{style=codesnippetstyle}
\begin{lstlisting}
; called code

target:     ...					; save RL
			... 				; allocate stack frame
			...					; save callee save
			
			...					; ??? work....				   <=====
			
			...					; do the function work
			
			...					; restore RL
			...					; restore callee saves
			...					: deallocate stack frame
			
								; ??? work....                 <=====
						
			BE (RL) 		    ; branch to return location	
\end{lstlisting}
\end{figure}

In addition to the work that needs to be done also for a local call, the external call, i.e. the cross region call, needs to do additional work. First the, since the target address is not in the same region, the virtual address needs to be obtained. Typically, during the load process, the region Id is assigned to the target region. The runtime features a \textbf{linkage table} for this purpose. The table will be explained the next section. 

Each module has global register, DP, which points to the module global data. A branch with the same file, the DP register remains unchanged. When calling code in a library, the DP value will change. Consequently, the entry and exit sequence of the called function potentially needs to save and restore the callers DP value.
