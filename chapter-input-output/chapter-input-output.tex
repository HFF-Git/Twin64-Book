\section*{Concepts}
\addcontentsline{toc}{section}{Concepts}

Twin-64 features a memory--mapped I/O architecture. A dedicated portion of the physical address space, specifically region zero,  is reserved exclusively for the I/O subsystem.  
This clear separation ensures predictable behavior during system initialization and simplifies both hardware design and software development.

The I/O architecture is organized around functional entities called \textbf{modules}. Typical modules include processors, memory controllers, and a variety of I/O cards. All modules are connected to the \textbf{system bus}, which is designed to support up to 64 modules in total.   Although real-world systems often provide fewer physical slots, the architecture itself imposes no such limitation.

During system startup, each module is assigned a dedicated 4 KByte physical address page. This page defines the base range to which the module responds and provides an fixed anchor for its control and status information.

Within each module, functionality is exposed through a collection of \textbf{registers}. These registers are accessible via standard load and store instructions, yet they are not required to be 64 bits wide. A core set of registers is common across all modules, ensuring a uniform interface for basic operations, while each module may additionally define its own device-specific register set to implement specialized features.

The registers found within a module’s hard physical address page are further subdivided. One portion forms the common supervisor register set, while the remainder is split among multiple \textbf{I/O element} register sets. An I/O element, also referred to as a \textbf{device}, represents the smallest addressable functional unit within a module. For example, a console terminal module may present two I/O elements: one dedicated to input and another dedicated to output.

The system bus itself spans a 256\,KB address space, aligned to a 256 KB boundary, with one page reserved for each possible module. In practice, physical constraints typically limit the number of installed modules, but the architecture accommodates growth by allowing several modules to reside on a single I/O card, each card occupying one physical slot on the bus.

Overall, the Twin-64 I/O subsystem aims to balance clarity, modularity, and extensibility.  
Its hierarchical structure, from modules to devices, and from hard physical pages to soft and broadcast regions, supports both straightforward configurations and more complex, densely populated systems.

The I/O address range is a 256Mbyte area in the first 4 Gbyte region, i.e. region 0. The area is divided into several ranges. The first range contains the processor dependent code. There is a set of library functions to manage the processor itself but also provide low-level primitives for the boot process and the operating system. 

Each module listens to three distinct address ranges:

\begin{itemize}
    \item \textbf{Hard physical address range:}  
    This is the module’s assigned physical page at system initialization. Its position is determined by the card’s physical slot and the module’s index on the I/O card. Access to the hard physical page is privileged and reserved for supervisor-level operations.

    \item \textbf{Soft physical address range:}  
    Certain modules require more than a single page of address space to support their full operational state. For such cases, additional memory pages may be dynamically allocated in the soft physical region, providing the necessary flexibility without compromising layout consistency.

    \item \textbf{Broadcast address range:}  
    This range provides a mirrored view of the entire system-bus address space. A write issued through a broadcast address updates the corresponding register in \emph{every} module on the bus. Broadcast writes are useful for system-wide resets, synchronization operations, or events that must be delivered simultaneously to all modules.
\end{itemize}


\section*{Hard physical address range}
\addcontentsline{toc}{section}{Hard physical address range}

The \textbf{hard physical address} range contains a I/O page for each I/O modules installed. There is room for 64 I/O modules. Each module provides a set of registers in that page. The first 16 registers are the 16 supervisory registers. Next are the I/O element registers. There are up to 62 I/O elements, each element providing up to 8 registers.

\section*{Broadcast address range}
\addcontentsline{toc}{section}{Broadcast address range}

The \textbf{broadcast address} range provides a mirrored view of the hard physical address space.  
Writes issued to broadcastable registers within this range are not directed to a single module.  
Instead, the write operation is applied simultaneously to all I/O modules on the system bus.  
This mechanism is typically used for system-wide commands such as resets, synchronization events,  
or configuration updates that must reach every module at once.

\section*{Soft physical address range}
\addcontentsline{toc}{section}{Soft physical address range}

In many cases, a module requires more addressable registers or data structures than can be  
accommodated within its hard physical address page. The \textbf{soft physical address} range provides a flexible region where additional space may be allocated for an I/O module’s extended functionality. A module responds to memory accesses within its configured soft physical address space, which may reside  either in the I/O address space or the general memory address space.

For example, a memory controller has a hard physical page in the I/O address space, yet its soft physical region typically encompasses all or part of the system’s physical memory. Similarly, a processor may allocate a soft physical address range within the I/O space to store performance counters or diagnostic information that needs to be accessible without interfering with normal memory traffic.
