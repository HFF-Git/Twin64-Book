Some general registers have a dedicated use. Register zero will always return a zero value when read and a write operation will have no effect. Although there are only 16 general registers in the CPU, implementing such a register greatly simplifies the instruction design, in that it for example provides a target when the result is not needed. General register one is a scratch register and also serves as an implicit target for some instructions. Other general registers may have a dedicated purpose defined in the runtime architecture conventions.

\section*{Control Registers}
\addcontentsline{toc}{section}{Control Registers}

Twin-64 has 16 defined control registers, numbered \texttt{CR0} to \texttt{CR15}. There are two instructions to move the contents of a control register to a general register and vice versa. A move to a control register will always move the entire 64-bit word. A move to a general register from a control register will move the size of the control register right aligned to the general register. The following table depicts the control registers defined for Twin-64.

\subsection*{Protection Id Registers}
\addcontentsline{toc}{subsection}{Protection Id Registers}

Twin-64 allows to record a set of up to 8 protection Id values in the processor control registers \texttt{CR4} to \texttt{CR7} which are accessible to the currently executing process. The protection Id is identical to a segment Id. For each access to a segment that has the protection check bit set, one of the protection Id control registers must match the segment Id. If not, a protection violation trap is raised. The rightmost bit of each of the eight PIDs is the write disable (W) bit. When the bit is set, that PID cannot be used to grant write access. See also the chapter on addressing and access control.

\subsection*{Interrupt Vector Address}
\addcontentsline{toc}{subsection}{Interrupt Vector Address}

The Interruption Vector Address (\texttt{IVA}) control register \texttt{CR8} contains the address of the base of an array of service procedures assigned to the interruption classes. The lower 14 bits of the IVA are reserved and set to zero. The address written to it must therefore be aligned on a page size.  The IVA is always located in the first segment, segment 0. The array of interruption service procedures is indexed by the interruption numbers described in the chapter on interrupts.

\subsection*{External Interrupt Enable Mask}
\addcontentsline{toc}{subsection}{External Interrupt Enable Mask}

The External Interrupt Enable Mask (\texttt{EIEM}) is stored in \texttt{CR15)}. It is 64-bit register containing a bit for each of the 64 external interrupts. A bit cleared in the EIEM masks external interrupts corresponding to that bit position.

\subsection*{Interruption PSW}
\addcontentsline{toc}{subsection}{Interruption PSW}

Upon an interrupt the interrupted instruction address and the processor status bits are saved to this control register. 

\subsection*{Interruption Argument Registers}
\addcontentsline{toc}{subsection}{Argument Registers}

The Interruption Argument Registers (\texttt{CR11} and \texttt{CR12}) are used to pass an instruction and a virtual address to an interruption handler. The values in these two control registers for each interruption class are described in the chapter on interrupts.

\subsection*{Scratch Registers}
\addcontentsline{toc}{subsection}{External Interrupt Register}

The control registers \texttt{CR13}, \texttt{CR14}, \texttt{CR15} are scratch registers used by the interrupt handlers or alike. \texttt{CR12} and \texttt{CR13} are privileged register only accessible from code running at priv level, \texttt{CR15} is accessible from any priv level.

\section*{Data Types}
\addcontentsline{toc}{section}{Data Types}

The fundamental data type is a 64-bit signed integer with the most significant bit being left and the least significant bit being right. All computation is  performed on 64-bit signed integer numbers. Memory access is performed on a double, word, half-word or byte basis. Word, Half and Byte data loaded is sign-extended. Store operations just store the corresponding value in memory as is.

\section*{Byte Ordering}
\addcontentsline{toc}{section}{Byte Ordering}

Twin-64 is internally a big-endian machine. It does however support little endian and big endian byte ordering models. 

The optional E-bit in the PSW register controls whether loads and stores use big endian or little endian byte ordering. When the E-bit is cleared, all larger-than-byte load and store operations are in big endian ordering. The lower addressed bytes in memory correspond to the higher-order bytes in the register. When the E-bit is set, all larger-than-byte load and store operations are in little endian mode. The lower-addressed bytes in memory correspond to
the lower-order bytes in the register. 

In addition, the E-bit also controls the instruction fetch. Processors that implement byte ordering, must also provide a default setting to be set by software. The default setting is stored in the processor configuration register \texttt{CR0}. The default endian bit setting controls whether the PSW E-bit is set to 0 or 1 on interruptions and also controls how to interpret the data in the address translation related tables.

\section*{Translation Lookaside Buffer}
\addcontentsline{toc}{section}{Translation Lookaside Buffer}

Virtual address translations are stored in a translation look-aside buffer. The TLB is essentially a copy of the page table entry information that represents the virtual page currently loaded at the physical address. Depending on the hardware implementation, there can be a combined TLB or a separate instruction TLB and data TLB. The virtual page number, VPN, is stored along with attributes and the physical page number, PPN, in the TLB entry. The current architecture allows for a 36-bit physical address space, which is a maximum of 64GB memory space.

The first segments, segment 0 to 15, are special in that they bypass the TLB entirely. A virtual address in the range 0x00\_0000\_0000 to 0x03\_FFFF\_FFFF directly maps to the physical address of the same range. The address range represents physical memory and can only be accessed in privileged mode. 

\section*{Caches}
\addcontentsline{toc}{section}{Caches}

Twin-64 is a cache visible architecture, featuring cache models for unified or separate instruction and data cache. While hardware directly controls the cache for cache line insertion and replacement, the software has explicit control on flushing and purging cache line entries. Twin-64 caches are virtually indexed, physically tagged caches. To speed up the entire memory access process, the cache uses the virtual address to index into the arrays, which can be done in parallel to the TLB lookup. When the physical address tag in the cache and the physical page address in the TLB match, there will be a cache hit. 

The Twin-64 cache architecture will not specify the size of cache lines or cache organization. The Twin-64 architecture does not specify a particular cache design and hierarchy model. The architecture just specifies instructions to handle cache operations, such as delete or flushing a cache.

The TLB entry contains information whether the data accessed should be cached or not. For segments 0 to 15, where the TLB is bypassed, the hardware decides on the caching option. The memory address ranges are always cached. The I7O address range is not cached.
